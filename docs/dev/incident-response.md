# 障害対応フロー

## 目的

- 不具合・回帰・セキュリティ問題の初動を標準化し、対応の遅延を防ぐ。

## 重大度

| 重大度 | 例 | 目標初動 |
|---|---|---|
| S0 | 起動不能、データ破壊、重大セキュリティ問題 | 24時間以内 |
| S1 | 主要機能が利用不可、対局続行不能 | 48時間以内 |
| S2 | 回避策ありの機能不具合 | 5営業日以内 |
| S3 | 軽微な表示不具合、改善要望 | 次回以降の計画で対応 |

## 受付

- 通常不具合: `bug` テンプレート
- 回帰: `regression` テンプレート
- セキュリティ: `security` テンプレート（重大案件は Security Advisories 推奨）

## トリアージ手順

1. 再現可否の確認（手順・ログ・環境情報）
2. 重大度の決定（S0-S3）
3. 影響範囲の特定（UI / 対局 / 棋譜 / 解析 / 配布）
4. 担当アサインと期限設定
5. 必要に応じて回避策を先出し

## 修正時の必須事項

- 回帰テスト追加、または既存テストの強化
- 影響範囲に応じた手動テストシナリオ実施
- リリースノートへの記載（ユーザー影響がある場合）

## クローズ条件

- 再現手順で再発しないことを確認
- 自動テストが通過
- 修正内容と検証結果を Issue に記録
